المشروع غني وظيفيًا وفيه عمل كبير على SEO والمحتوى العربي، لكن حالته الحالية ليست جاهزة كنسخة نهائية خالية من العيوب. السبب: تداخل أكثر من منتج داخل نفس المستودع، بيئة اختبار غير مستقرة، ملفات مكررة/يتيمة، وتضارب واضح في مسارات النشر والحوكمة.

  • الاختبارات على الجذر فشلت: 5 حالات Groq + 2 Suites غير قادرة على التحميل.
  • سبب فشل Groq الأساسي: الاختبار يستدعي /api/groq/stream بينما السيرفر يدعم /api/ai/stream.
  • أوامر brightai-platform فشلت بسبب عدم توفر react-scripts (اعتماديات غير منصبة في هذا المسار).
  • فحص SEO الحالي نجح: 18/18 صفحات، وفحص الروابط/الموارد أعطى 0 أعطال.

  • ملفات مكررة (تطابق hash فعلي) موجودة في backend/frontend/assets.

4) المشاكل الحرجة (تمنع اعتبار المشروع «نهائي»)

  1. تعطّل مسار الاختبارات: test setup الحالي يخلط بين مشاريع متعددة داخل الريبو بدون عزل.
  2. اختبار/مسار API غير متطابق: backend/groq-stream.functional.test.js مقابل backend/server.js.
  3. ازدواجية تشغيلية: وجود نسخ غير مستخدمة مثل backend/routes/search 2.js وsearch 3.js وfrontend/js/search 2.js.
  4. تضارب بيئة النشر: وجود netlify.toml + .htaccess + _headers (Cloudflare style) في نفس الوقت بدون policy موحّدة.
  5. توثيق غير مطابق للواقع: README يشير إلى workflow غير موجود فعليًا في .github/workflows.

5) المشاكل المهمة (يجب إصلاحها في الدورة الحالية)

  1. فجوات SEO جزئية: canonical/description/H1 ناقصة في مجموعة صفحات (خصوصًا docs/offline وبعض blogger).
  2. قبول مفتاح API من Body/Header في بعض مسارات Groq يحتاج policy صريحة (BYOK) مع تقييد ومراقبة.
  3. استخدام واسع لـ innerHTML في صفحات تفاعلية كثيرة؛ يلزم توحيد sanitization قبل الحقن.
  4. ملف sw.js غير مسجل في واجهة static الحالية (يتطلب قرار: تفعيل فعلي أو إزالة).
  5. لا يوجد brightai-platform/.env.example رغم الحاجة التشغيلية.

6) فرص التحسين (Optimization Queue)

  • تقليل ضخامة الصفحات: صفحات كثيرة >100KB HTML، وهذا يضغط LCP/TTFB/التصيير على الأجهزة الضعيفة.
  • توحيد الأصول المكررة (fonts/icons) لتقليل العبء وإزالة الالتباس.
  • رفع حوكمة CI بإضافة pipeline إلزامي: test + SEO + budget + duplicate/orphan scan.

8) الخلاصة التنفيذية

المشروع قوي من ناحية المحتوى وتغطية الصفحات، لكن قبل اعتباره إصدارًا نهائيًا يجب تنفيذ دورة تنظيف وضبط حوكمة كاملة. أولوية التنفيذ: تثبيت الاختبارات، إزالة التكرارات/الملفات اليتيمة، توحيد سياسة النشر، ثم إقفال فجوات الأمن والـSEO الجزئية.

القسم الثاني: Prompts جاهزة لـ Codex (تنفيذ مباشر)

Prompt 1 — تثبيت البنية الأساسية للمونوريبو

افحص المشروع كـ monorepo من 3 وحدات (frontend, backend, brightai-platform)، ثم: 1) أنشئ strategy واضحة للتشغيل والاختبار لكل وحدة، 2) عدل scripts الجذر لتشغّل الوحدات بعزل، 3) امنع تداخل الاختبارات بين الوحدات. المطلوب النهائي: README موحّد + scripts موثوقة + أمر test واحد ناجح أو موضح سبب الفشل لكل وحدة.

Prompt 2 — إصلاح فشل اختبارات Groq وتوافق المسارات

أصلح عدم التوافق بين الاختبارات والسيرفر لمسار البث: إما دعم alias /api/groq/stream في backend/server.js أو تحديث الاختبارات إلى المسار الرسمي /api/ai/stream. ثم شغّل الاختبارات وتأكد أن حالات: success, runtime key, upstream error, timeout, rate-limit كلها Passing. أضف اختبار regression يمنع كسر هذا التوافق مستقبلًا.

Prompt 3 — تنظيف النسخ المكررة في backend/frontend

احذف النسخ غير المستخدمة التالية بعد التحقق من عدم وجود imports/refs: backend/routes/search 2.js, backend/routes/search 3.js, backend/utils/cache 2.js, frontend/js/search 2.js, backend/data/search-index 2.json. ثم نفّذ scan يثبت عدم وجود مراجع مكسورة. أخيرًا حدّث أي docs تشير للملفات القديمة.

Prompt 4 — توحيد النشر والسياسات (Netlify/Apache/Headers)

راجع التضارب بين netlify.toml و.htaccess و_headers، وحدد Provider رسمي واحد للإنتاج. إذا Netlify هو الرسمي: انقل كل قواعد redirect/headers المهمة إلى netlify فقط، واجعل الملفات الأخرى للأرشفة/التوثيق. الهدف: source of truth واحد للـ routing/security headers/caching.

Prompt 5 — إقفال فجوات SEO الجزئية

أنشئ سكربت إصلاحي + تحقق تلقائي لكل صفحات HTML للتأكد من: title + meta description + canonical + H1 + hreflang الملائم. عالج الصفحات الناقصة تلقائيًا (خصوصًا docs/offline/blogger المخصصة)، ثم أعد تشغيل npm run sitemap:all وnpm run seo:check مع تقرير قبل/بعد.

Prompt 6 — تقوية الأمن (Hardening)

طبّق سياسة أمنية موحدة: 1) تقييد CORS بقائمة domains بيئية، 2) فصل وضع BYOK لمفاتيح Groq عن وضع server-managed مع feature flag وتدقيق request logging آمن، 3) إضافة sanitization layer موحدة لكل نقاط innerHTML الديناميكية، 4) رفع CSP أكثر صرامة تدريجيًا مع مراقبة كسر الواجهة. سلّم قائمة مخاطر قبل/بعد.

Prompt 7 — تحسين الأداء البنيوي

فكك الصفحات الثقيلة جدًا (خاصة index.html وصفحات interview/blog الكبيرة): انقل السكربتات inline إلى ملفات modular، قلل DOM المبدئي، واعتمد lazy-loading للأقسام الثقيلة. حدّث performance-budget ليقيس metrics واقعية إضافية (HTML max size, critical path JS/CSS, large page alerts).

Prompt 8 — توثيق وتشغيل الفريق

أنشئ توثيق تشغيل فعلي بدل النصوص العامة: كيفية setup لكل وحدة، env vars المطلوبة، أوامر الاختبار/البناء، وسياسة release/rollback. حدّث brightai-platform/README.md من قالب CRA إلى وثيقة منتج حقيقية. أضف brightai-platform/.env.example مع شرح القيم.

Prompt 9 (الأخير) — حذف الملفات غير المفيدة + دمج المكررات بدون تأثير

نفّذ عملية تنظيف نهائية آمنة على مرحلتين: (أ) حذف كل ملف غير مستخدم/مكرر مثبت بالمسح المرجعي والـhash، (ب) دمج الأصول المكررة (fonts/icons/blog variants) إلى مصدر واحد canonical مع redirects/refs محدثة. الشروط الإلزامية قبل التسليم: 1) لا كسر روابط داخلية أو موارد، 2) اجتياز اختبارات backend + فحص SEO + budget, 3) تقرير before/after يثبت عدد الملفات المحذوفة، ما دُمج، وما الأثر على الحجم والجودة.

ملاحظة تنفيذية: الترتيب الموصى به هو من Prompt 1 إلى Prompt 9 بالتسلسل. لا تبدأ الحذف النهائي قبل تثبيت الاختبارات والمرجعيات.

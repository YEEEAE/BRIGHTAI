القسم الثاني: Prompts جاهزة لـ Codex (مع النسخ الذكي)

✨ Prompt 1 — تثبيت البنية الأساسية للمونوريبو

افحص المشروع كـ monorepo من 3 وحدات (frontend, backend, brightai-platform)، ثم: 1) أنشئ strategy واضحة للتشغيل والاختبار لكل وحدة، 2) عدل scripts الجذر لتشغّل الوحدات بعزل، 3) امنع تداخل الاختبارات بين الوحدات. المطلوب النهائي: README موحّد + scripts موثوقة + أمر test واحد ناجح أو موضح سبب الفشل لكل وحدة.

✨ Prompt 2 — إصلاح فشل اختبارات Groq

أصلح عدم التوافق بين الاختبارات والسيرفر لمسار البث: إما دعم alias /api/groq/stream في backend/server.js أو تحديث الاختبارات إلى المسار الرسمي /api/ai/stream. ثم شغّل الاختبارات وتأكد أن حالات: success, runtime key, upstream error, timeout, rate-limit كلها Passing. أضف اختبار regression يمنع كسر هذا التوافق مستقبلًا.

✨ Prompt 3 — تنظيف النسخ المكررة

احذف النسخ غير المستخدمة التالية بعد التحقق من عدم وجود imports/refs: backend/routes/search 2.js, backend/routes/search 3.js, backend/utils/cache 2.js, frontend/js/search 2.js, backend/data/search-index 2.json. ثم نفّذ scan يثبت عدم وجود مراجع مكسورة. أخيرًا حدّث أي docs تشير للملفات القديمة.

✨ Prompt 4 — توحيد النشر والسياسات

راجع التضارب بين netlify.toml و.htaccess و_headers، وحدد Provider رسمي واحد للإنتاج. إذا Netlify هو الرسمي: انقل كل قواعد redirect/headers المهمة إلى netlify فقط، واجعل الملفات الأخرى للأرشفة/التوثيق. الهدف: source of truth واحد للـ routing/security headers/caching.

✨ Prompt 5 — إقفال فجوات SEO

أنشئ سكربت إصلاحي + تحقق تلقائي لكل صفحات HTML للتأكد من: title + meta description + canonical + H1 + hreflang الملائم. عالج الصفحات الناقصة تلقائيًا (خصوصًا docs/offline/blogger المخصصة)، ثم أعد تشغيل npm run sitemap:all و npm run seo:check مع تقرير قبل/بعد.

✨ Prompt 6 — تقوية الأمن (Hardening)

طبّق سياسة أمنية موحدة: 1) تقييد CORS بقائمة domains بيئية، 2) فصل وضع BYOK لمفاتيح Groq عن وضع server-managed مع feature flag وتدقيق request logging آمن، 3) إضافة sanitization layer موحدة لكل نقاط innerHTML الديناميكية، 4) رفع CSP أكثر صرامة تدريجيًا مع مراقبة كسر الواجهة. سلّم قائمة مخاطر قبل/بعد.

✨ Prompt 7 — تحسين الأداء البنيوي

فكك الصفحات الثقيلة جدًا (خاصة index.html وصفحات interview/blog الكبيرة): انقل السكربتات inline إلى ملفات modular، قلل DOM المبدئي، واعتمد lazy-loading للأقسام الثقيلة. حدّث performance-budget ليقيس metrics واقعية إضافية.

✨ Prompt 8 — توثيق وتشغيل الفريق

أنشئ توثيق تشغيل فعلي بدل النصوص العامة: كيفية setup لكل وحدة، env vars المطلوبة، أوامر الاختبار/البناء، وسياسة release/rollback. حدّث brightai-platform/README.md من قالب CRA إلى وثيقة منتج حقيقية. أضف brightai-platform/.env.example مع شرح القيم.

✨ Prompt 9 — التنظيف النهائي

نفّذ عملية تنظيف نهائية آمنة على مرحلتين: (أ) حذف كل ملف غير مستخدم/مكرر مثبت بالمسح المرجعي والـhash، (ب) دمج الأصول المكررة (fonts/icons/blog variants) إلى مصدر واحد canonical مع redirects/refs محدثة. الشروط الإلزامية قبل التسليم: 1) لا كسر روابط داخلية، 2) اجتياز اختبارات backend وفحص SEO، 3) تقرير before/after.

🚀 Prompt 10 — إصلاح الاختبارات القديمة

عالج مشاكل CI/test legacy الموجودة حاليًا حتى لو كانت خارج نطاق الميزة الحالية: 1) ثبّت اعتماديات الاختبار المفقودة لكل وحدة (backend و brightai-platform) عبر setup واضح ومعزول. 2) أصلح مشكلة اختبارات backend التي ترجع 404 بسبب عدم تطابق المسارات (مثل /api/groq/stream مقابل /api/ai/stream) بإضافة alias متوافق أو تحديث الاختبارات مع الحفاظ على backward compatibility. 3) حدّث ملفات إعداد الاختبار لتجنب تداخل مشاريع monorepo في نفس runner. 4) نفّذ اختبار end-to-end minimal يثبت أن مسارات البث الرئيسية ترجع 200 في سيناريو النجاح. 5) سلّم تقريرًا نهائيًا يفرق بين: أعطال مرتبطة بالميزة الحالية وأعطال legacy تم إصلاحها الآن.

🚀 Prompt 11 — ربط Gemini الفعلي بالنشر

أصلح مشكلة اتصال Gemini في بيئة الإنتاج عبر توحيد طبقة API Gateway بين الواجهة والخادم: 1) اعتمد متغيرًا موحدًا لقاعدة الـ API مثل BRIGHTAI_API_BASE (مع fallback آمن إلى نفس الأصل فقط عند توفر backend فعلي). 2) اجعل جميع نداءات الواجهة التي تستخدم /api/gemini/chat و /api/gemini/chat/stream تمر عبر دالة مركزية تبني URL الإنتاج الصحيح بدل hardcoded paths. 3) وحّد استراتيجية النشر: - إذا الإنتاج على Netlify: فعّل redirect /api/* -> /.netlify/functions/all كمسار رسمي. - إذا الإنتاج على Apache/Cloudflare: أنشئ proxy/rewrite مكافئ أو endpoint backend منفصل واضح (مثل https://api.brightai.sa) مع CORS مضبوط. 4) أضف فحص preflight عند بدء التطبيق على الإنتاج يستدعي /api/health أو endpoint health مكافئ ويعرض رسالة تشخيص واضحة إذا فشل الربط (بدل فشل صامت). 5) تحقق من متغيرات البيئة في النشر (GEMINI_API_KEY, GEMINI_MODEL) وتأكد أن 503 بسبب key missing يظهر برسالة تشغيلية مفهومة. 6) سلّم smoke test على رابط الإنتاج الفعلي يثبت نجاح: POST /api/gemini/chat و POST /api/gemini/chat/stream من المتصفح بدون 404/500.

أوامر تحسين إضافية (Prompts إضافية جاهزة للنسخ)

💡 حذف التبعيات وتنظيف المستودع

استخدم Codex لإزالة جميع مجلدات node_modules من المشروع وتحديث ملف .gitignore لاستبعادها. كذلك، احذف مجلد __MACOSX وأي ملفات تقارير أو وثائق داخلية (مثل .docx أو .md) غير ضرورية لتشغيل الموقع لتقليص حجم المستودع.

💡 تحديث هيكل النشر (Netlify) وتحديث الروابط

اضبط إعدادات Netlify (في netlify.toml) لجعل مجلد "frontend" هو الجذر المُنشر بدل الجذر الحالي. ثم استخدم Codex لفحص جميع الروابط الداخلية (anchor وCSS) وتعديل مسارات /frontend/... لتصبح نسبية وصحيحة على البنية الجديدة بدون كسر أي مورد.

💡 دمج ملفات CSS وإزالة التكرار

حدد واطلب دمج الملفات style.css المكررة أو غير المرغوب فيها في ملفات CSS رئيسية (مثل دمج main.bundle.css مع bundle-pages.css إذا أمكن) لتقليل التكرار. تَحقَّق من أن استيراد الخطوط والأنماط الأساسي يتم في ملفات عامة بدلاً من نسخ متشابهة.
ملاحظة تنفيذية حاسمة: الترتيب الموصى به هو من Prompt 1 إلى Prompt 11 بالتسلسل. لا تبدأ بمهام الحذف النهائي قبل إرساء الاختبارات والمرجعيات. بعد التنفيذ، اختبر الـ API والـ Console في بيئة النشر للتأكد من خلو النظام من أي أخطاء (404/500).
